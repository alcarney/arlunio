The Setup.py File¶

The setup.py file is what the packaging tools use to determine what should go into stylo when it is packaged as well as define any metadata about the project itself.

Here is the contents of that file

from setuptools import setup, find_packages

from stylo import __version__


def readme():
    with open("README.md") as f:
        return f.read()


setup(
    name="stylo",
    version=__version__,
    description="Using a blend of Python and Maths for the creation of images",
    long_description=readme(),
    long_description_content_type="text/markdown",
    classifiers=[
        "Development Status :: 3 - Alpha",
        "License :: OSI Approved :: MIT License",
        "Programming Language :: Python :: 3.5",
        "Programming Language :: Python :: 3.6",
        "Programming Language :: Python :: 3.7",
        "Programming Language :: Python :: 3 :: Only",
        "Topic :: Multimedia :: Graphics",
    ],
    author="Alex Carney",
    author_email="alcarneyme@gmail.com",
    license="MIT",
    packages=find_packages(".", exclude="tests"),
    install_requires=["Pillow", "hypothesis", "matplotlib", "numpy", "pytest", "tqdm"],
    python_requires=">=3.5",
    setup_requires=["pytest-runner"],
    test_suite="tests",
    include_package_data=True,
    zip_safe=False,
)

Package Metadata¶

Name, Version, Description¶

Some standard info about the project, such as its name. The version is set to the value of the __version__ string from the stylo/_version.py file so that it is always up to date. There is also a one line description about what stylo does.

There is also a long_description field, this is what will be displayed on the project page on PyPi this is generally a project’s README file. Since the setup.py is just regular python code we write a readme() function to read the contents of the README.rst file so that this field is always up to date.

Authors, License, Classifiers¶

The author and author_email fields give you some idea about the people behind the project and a way to get in touch.

The license field tells you what license stylo is available under and dictates what you can and can’t do with the code. stylo is available under the MIT License and since I’m not a lawyer I recommend you follow the link if you want to know what the terms of the license are all about.

Then we have a number of classifiers, these are tags that help indicate the current state of the project. This can range from anything from the versions of Python that are supported to the license the project is available under. There are many classifiers to choose from here is a full list for those who are interested.

Note

There needs to be a conversation about when people are considered to be “core” contributors and therefore should be added to the authors list.

Packages & Requirements¶

Here we get into what is included in the package, we use the find_packages function to automatically discover the python files inside the stylo folder that need to be included.

We also need to specify our dependencies in the install_requires field so that when people pip install stylo all of the required packages are discovered and installed.

Those with a keen eye should notice that packages like hypothesis and pytest are not typically installation requirements as they are testing tools and only required by developers. Normally you would be right, however stylo does distribute a number of tests as part of the stylo.testing package as part of our extension guarantee. Since these packages are required to import objects from this package we include them as installation dependencies.

Finally the python_requires field specifies the minimum version that we support.

Miscellaneous Fields¶

The setup-requires and test_suite fields are a holdover from previous versions of stylo where the tests were run using the command python setup.py test. I am not sure if they are required anymore.

I am not sure what the zip_safe and include_package_data fields are for.
